## Overview

The v0.3.0 release introduces `RecurPixel.Notify.Dashboard` — a new observability package for tracking and troubleshooting notification delivery. The package follows a **data-first approach**, prioritizing data integrity and API design over UI.

**Key Principle:** UI is built last. The data layer must be solid and verified before any HTML or client code is written.

## Build Order (Critical)

### Phase 1: Data Layer & Orchestrator Integration

**Deliverables:**
- `NotificationLog` entity class
- `INotificationLogStore` interface
- SQL Server implementation of `INotificationLogStore`
- Update `Orchestrator.TriggerAsync` and `BulkTriggerAsync` to use `OnDelivery` hook
- Unit tests for log writing workflow

**Why First:**
- Everything downstream depends on accurate data
- Easier to debug data issues early
- UI bugs can be hidden by bad data; won't have that excuse if data is proven solid

**Key Decisions:**
- Should `NotificationLog` include full provider response body? (Yes, for debugging; use TEXT/JSONB column)
- Partition strategy for large datasets? (Recommend by date for older logs)
- Retention policy? (Defer to v0.3.1, but design schema to support it)

**Acceptance Criteria:**
- Manual test: trigger 10 notifications, verify all logged with correct timestamps, channels, statuses
- Unit test: mock store is called for every delivery result
- Integration test: real SQL Server logs a batch of 50 notifications correctly

---

### Phase 2: BulkBatchId Grouping

**Deliverables:**
- Modify `BulkTriggerAsync` to generate `BulkBatchId` (ULID or Guid)
- Pass `BulkBatchId` through each `NotifyResult` returned from trigger
- Update `NotificationLog` entity to include `BulkBatchId` column
- Unit tests for batch ID propagation

**Why This Phase:**
- Small, isolated change in one method
- Critical for dashboard grouping (users want to see "this bulk send" as a unit)
- Better to validate the plumbing now before UI work
- Does not touch adapter code or other areas

**Orchestrator Change:**
```csharp
public async Task<BulkTriggerResult> BulkTriggerAsync(
    IEnumerable<NotifyContext> contexts,
    CancellationToken cancellationToken = default)
{
    var batchId = Ulid.NewUlid().ToString(); // NEW
    var results = new List<TriggerResult>();
    
    foreach (var context in contexts)
    {
        var result = await TriggerAsync(context, cancellationToken);
        result.BulkBatchId = batchId; // NEW: attach batch ID to each result
        results.Add(result);
    }
    
    return new BulkTriggerResult { Results = results };
}
```

**Acceptance Criteria:**
- Every `NotifyResult` from bulk send contains the same `BulkBatchId`
- Single send (non-bulk) has null or empty `BulkBatchId`
- All logs for a batch query are linked by `BulkBatchId`
- Existing tests pass without modification

### Phase 3: REST API Layer

**Deliverables:**
- MVC/Minimal API endpoints in Dashboard package
- Query logs by date range, channel, status, recipient
- Aggregate endpoints: delivery success rate, channel breakdown, failure histogram
- Batch endpoints: get all logs for a `BulkBatchId`
- Implement pagination and filtering

**Endpoints (Examples):**
- `GET /api/notifications/logs` — all logs (paginated)
- `GET /api/notifications/logs?channel=email&status=failed&from=2026-03-01` — filtered
- `GET /api/notifications/batches/{batchId}` — all notifications in a batch
- `GET /api/notifications/channels` — per-channel status summary
- `GET /api/notifications/aggregate` — success rate, retry count, etc.

**Why This Phase:**
- Data layer is validated; now expose it safely
- API is testable independently of UI
- Can manually test with Postman/curl before UI is written

**Acceptance Criteria:**
- All CRUD + filter endpoints return 200 and correct JSON
- Pagination works (limit, offset, total count)
- Timestamp filtering returns correct date range
- Performance: queries on 1M logs complete in <2 seconds

---

### Phase 4: Embedded HTML Dashboard

**Deliverables:**
- Single-page dashboard HTML/CSS/JavaScript (minimal dependencies)
- Real-time logs table with sort, filter, search
- Channel status breakdown (pie/bar chart)
- Failure histogram (hourly failures over last 7 days)
- Retry history for individual notification
- Live send testing (trigger test notification, watch it log in real-time)

**Why Last:**
- Without solid data + API, the UI just hides problems
- Developers can already use JSON API; UI is a convenience layer
- Easier to iterate on UI once backend is stable
- Reduces scope creep (UI can always be improved post-release)

**Design Principles:**
- No external dependencies (Bootstrap, jQuery, React, etc.)
- Embed as static HTML/CSS in `RecurPixel.Notify.Dashboard` assembly
- Serve from a single Razor view or static file + API routes
- Auth: inherit from parent app's auth context (use existing [Authorize])

**Acceptance Criteria:**
- Dashboard loads and authenticates via parent app's middleware
- Logs table shows last 100 deliveries with timestamps, channels, statuses
- Search/filter works (by recipient, channel, date)
- Click notification expands to show provider response
- Click batch ID shows all notifications in batch
- Performance: page loads <1 second with 10K logs

## Integration with Core

### 1. OnDelivery Hook

The Dashboard hooks into the existing `OnDelivery` callback:

```csharp
// In app startup
services
    .AddRecurPixelNotify(notifyOptions, orchestratorOptions)
    .AddNotificationDashboard()
    .UseNotificationLogging(new SqlServerNotificationLogStore(connectionString));
```

Behind the scenes:
```csharp
orchestratorOptions.OnDelivery += async (result) =>
{
    // Dashboard captures every delivery result
    await logStore.LogDeliveryAsync(
        new NotificationLog
        {
            BulkBatchId = result.BulkBatchId,
            Channel = result.Channel,
            Status = result.Status,
            Recipient = result.To,
            ProviderResponse = result.ProviderResponse,
            Timestamp = DateTime.UtcNow,
            // ...
        },
        cancellationToken
    );
};
```

### 2. No Breaking Changes to Core

- Core API unchanged
- Dashboard is optional (users can opt-in via `AddNotificationDashboard`)
- Adapters untouched

## Related Issues / Decisions

1. **BulkBatchId Format:** ULID recommended (sortable, URL-safe, shorter than Guid)
2. **Log Retention:** Default unlimited; applications should implement cleanup via INotificationLogStore
3. **UI Framework:** Vanilla JS only (no dependencies) to keep Dashboard package lightweight
4. **Auth:** Inherit from parent app; Dashboard doesn't add its own auth layer
